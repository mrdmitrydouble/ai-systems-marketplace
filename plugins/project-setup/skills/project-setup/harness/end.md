Пользователь запустил `/end` — начинается протокол закрытия сессии.

Выполни **СТРОГО В ЭТОМ ПОРЯДКЕ**. Не пропускай шаги. Не объявляй «сессия закрыта» до финального PASS.

## Шаг 0 — Слей инбоксы параллельных сессий

Если в `Система/память/inbox_sessions/` есть `*.md` — это дельты сессий, работавших параллельно (страж уводит их записи туда вместо общих файлов). Прочитай каждый, влей содержимое по назначению (заголовки «## для X» указывают целевой файл), удали файл-инбокс. Только после этого обновляй память.

## Шаг 1 — Обнови файлы памяти

Если эти файлы не обновлены в текущей сессии — обнови сейчас по факту того, что было сделано:

1. **`Система/память/PROGRESS.md`** — новая запись о сессии:
   - Дата, номер сессии (из HANDOFF + 1)
   - Что сделано (ключевые действия, коммиты)
   - Решения
   - Обязательства (🔴🟠🟡)
   - Следующие шаги
2. **`Система/память/SYSTEM_LOG.md`** — минимум 1 наблюдение (удача/ошибка/идея/паттерн). Если сессия была содержательной — больше.
3. **`Система/память/CHANGELOG.md`** — если менялись системные файлы (CLAUDE.md, RULES.md, settings.json, хуки, скрипты).
4. **`Система/память/HANDOFF.md`** — полный rewrite:
   - Номер и дата сессии
   - Как запустить следующую сессию (одна команда)
   - Что сделано (основные блоки)
   - Открытые критичные пункты (🔴🟠🟡)
   - Остаются в наблюдении
   - Коммиты сессии
   - Паттерны сессии / главный урок
5. **`Система/память/TRACKER.md`** — если закрыты/открыты задачи.

## Шаг 2 — Запусти валидацию

```bash
bash Система/scripts/close_session_check.sh
```

## Шаг 3 — Разбор результата

- **PASS** → переходи к Шагу 4.
- **FAIL** → прочитай вывод, исправь указанные блокеры, повтори Шаг 2. НЕ декларируй завершение.
- **FAIL из-за чужой live-сессии** (страж памяти отклонял записи в PROGRESS/SYSTEM_LOG/HANDOFF, уводя в `inbox_sessions/`): это не блокер. Запиши wrap-up-дельту в `Система/память/inbox_sessions/` (заголовки «## для PROGRESS» и т.п.), сделай commit+push своей дельты, сообщи «закрыто в инбокс — финальное слияние сделает последняя сессия». Полный wrap-up + слияние выполнит та сессия, что закроется последней.

## Шаг 4 — Git commit + push

```bash
# Добавляй ЯВНЫЕ файлы (не git add -A — в team иначе захватишь чужой WIP):
git add Система/память/*.md CLAUDE.md   # + остальные изменённые тобой файлы
git status  # покажи пользователю что коммитим
```

Коммит с осмысленным сообщением (не «wrap-up», а по сути сделанного):
```bash
git commit -m "Сессия N: <суть> + wrap-up"
git push origin main   # имя ветки/remote — по проекту
```

## Шаг 5 — Маркеры закрытия

```bash
rm -f .claude/.session_incomplete.json   # если был recovery — маркер обработан, убрать (иначе RECOVERY-петля)
touch .claude/.session_closed.flag       # сигнал Stop/SessionEnd-хукам: закрытие легитимное, не вмешиваться
```

Если правка `.session_incomplete.json` вошла в этот же коммит — добавь `git rm` и допиши в коммит.

## Шаг 6 — Отчитайся пользователю

Одно сообщение (≤3 строк):
```
Сессия N закрыта. Коммит <hash>, push ok.
Главный итог: <одна строка>.
Следующий фокус: <из HANDOFF>.
```

---

## Если что-то пошло не так

- Коммит не прошёл из-за pre-commit hook — **не делать `--no-verify`**. Разобраться в причине, починить, создать новый коммит.
- Push отклонён (non-fast-forward) — `git pull --rebase origin main`, разрешить конфликты, повторить push.
- Валидация FAIL после нескольких попыток — опиши пользователю что не сходится, спроси совет. Не игнорируй.
