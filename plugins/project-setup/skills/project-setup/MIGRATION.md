# MIGRATION.md — Протокол миграции проектов (v8)
> Загружается ТОЛЬКО при миграции.
> Архитектура: инвариантное ядро (всегда) + адаптивный слой (генерируется под систему).

⚠️ **Пре-чеклист мажорной миграции: сначала стабилизация, потом фичи.** Перед миграцией на новую мажорную версию — закрыть долг гигиены: валидатор/quick_health зелёный, реестры свежие (FILE_INDEX, TRACKER), память в лимитах. Мажорный скачок поверх нестабилизированной базы = системный риск (боевое Набл. 62).

---

## КОГДА ЧИТАТЬ ЭТОТ ФАЙЛ

- Пользователь говорит «мигрируй», «обнови структуру», «апгрейд на v[X]»
- VERSION.md показывает `needs_update: true`
- Версия скилла в системе отличается от установленной
- Любые структурные изменения (переименование папок, перемещение файлов)

---

## ИНВАРИАНТНОЕ ЯДРО

> Эти правила применяются ВСЕГДА, к ЛЮБОЙ миграции. Адаптивный слой может ДОБАВЛЯТЬ шаги, но НИКОГДА не убирает инвариантные.

### Инвариант 1 — Бэкап до старта

Миграция НЕ НАЧИНАЕТСЯ без подтверждённого бэкапа.

Агент говорит:
> «Перед миграцией создай бэкап проекта. Варианты:
> — zip-архив всего проекта → положи в Sources/
> — [Claude Code] git commit + git tag pre-migration-[date]
> Скажи когда готово.»

Без ответа «готово» → не продолжать.

### Инвариант 2 — Discovery перед планом

Агент ОБЯЗАН проанализировать систему перед планом:
1. Прочитать CLAUDE.md, RULES.md, ARCH_PRINCIPLES.md, VERSION.md
2. Просканировать файловую структуру
3. Определить: горячие (каждую сессию), тёплые (по запросу), холодные (архив)
4. Собрать reference audit: grep всех паттернов, которые будут меняться — **ВКЛЮЧАЯ скрытые слои, которые grep проекта НЕ ловит** (боевой класс LOOV-C3 / АвтоГИС, с.47: путь проекта дублируется в ~6 слоях, часть — app-внутренние): промпты scheduled-задач (`~/.claude/scheduled-tasks/*/SKILL.md`) **И их cwd-биндинг** в app `scheduled-tasks.json`; `claude_desktop_config.json` (auto-acks + trusted-folders); `git-worktrees.json`; PROJECT_PATHS.md; память. Обновить ВСЕ слои → иначе тихий отказ (планировщик молча skip «cwd no longer exists», промпт обновлён — задача мертва). После миграции прогнать dead-path-детектор (quick_health).
5. Результат → checkpoint-файл

### Инвариант 3 — Frozen zones

При миграции НИКОГДА не модифицировать:

| Зона | Почему |
|------|--------|
| `*_backup*`, `*_bak*` | Бэкапы |
| `Sources/*` | Входные данные неизменяемы |
| `Archive/*` | Историческая запись |
| `Documents/Final/*` | Доставлены получателю |
| Записи в логах до текущей даты | История |
| `CONSCIOUSNESS.md` | Только владелец редактирует |

Не уверен, является ли файл frozen → считать frozen.

### Инвариант 4 — Разделение фаз STRUCTURE и REFERENCES

Сначала переместить ВСЕ файлы. Потом обновить ВСЕ ссылки. Не смешивать.

### Инвариант 5 — Семантическая верификация

После обновления ссылок:
```
grep -r "СТАРЫЙ_ПАТТЕРН" [active_папки] --include="*.md"
```
0 совпадений в active-файлах = OK. Иначе → вернуться и исправить.

### Инвариант 6 — Frozen integrity check

Frozen-файлы НЕ ИЗМЕНИЛИСЬ? Сравнить размеры с checkpoint.

### Инвариант 7 — Root Cause Escalation при ошибке

```
1. Найди ошибку (E)
2. Найди системную причину (P)
3. P привела к другим ошибкам?
4. Мета-причина (MP)?
5. Повторяй до корня
6. Зафиксируй в SYSTEM_LOG
7. Для каждого уровня предложи исправление
```

### Инвариант 8 — Audit trail

Каждая миграция оставляет след:
- `System/working/migration_[date]/checkpoint.md` — состояние ДО
- `System/working/migration_[date]/operations.log` — что сделано
- `System/working/migration_[date]/verification.md` — результаты проверки
- Запись в SYSTEM_LOG, PROGRESS, CHANGELOG

## АДАПТИВНЫЙ СЛОЙ

> Генерируется агентом ПОД КАЖДУЮ КОНКРЕТНУЮ МИГРАЦИЮ.

### Фаза A — Discovery & Analysis

1. Структурный скан (все системные файлы, карта зависимостей)
2. Классификация: ACTIVE (обновлять) / FROZEN (не трогать) / HISTORICAL (не менять старое)
3. Оценка рисков (критичные пути, scheduled tasks, кросс-проектные ссылки)
4. Специфика проекта (нестандартные папки, расширения правил)

### Фаза B — Поиск свежих практик

Веб-поиск (не более 3 минут):
- Находка конфликтует с инвариантом → ОТКЛОНИТЬ
- Находка дополняет → ДОБАВИТЬ как рекомендацию
- Нет интернета → продолжать без этой фазы

### Фаза C — Plan & Confirm

Агент генерирует план:
```markdown
# План миграции: [проект] v[X] → v[Y]
## Что меняется
## 1. Структурные изменения
## 2. Обновление ссылок [N в M файлах]
## 3. FROZEN (НЕ трогаем)
## 4. Новые файлы
## 5. Scheduled tasks
## 6. [Claude Code] Git операции
## Развилки (вопросы пользователю)
```

Показать в чат → ждать «да» или правки.

### Фаза D — Execute

**D.1 — Structure:** создать папки, переместить файлы, логировать.
**D.2 — References:** для ACTIVE → заменить. FROZEN → SKIP. HISTORICAL → SKIP.
**D.3 — Scheduled tasks:** в промптах задач прописать НОВЫЙ абсолютный путь к проекту (рабочая папка задачи через update-tool не меняется — путь живёт в промпте; иначе задачи месяцами бьют по мёртвым путям, боевой случай — 6 зомби). 
**D.4 — Дополнительно:** VERSION.md, новые файлы. [team] Проект был в паузе 2+ недели → ритуал закрытия спящего режима (устаревшие 🆕 → ✅ с комментарием).
**D.5 — [Claude Code]:** git add + git commit с описанием миграции.

### Фаза E — Verify

**E.1 — Структурная:** папки существуют? Файлы на месте? Нет файлов 0 байт?
**E.2 — Семантическая:** grep=0 для старых паттернов в active.
**E.3 — Frozen integrity:** размеры = checkpoint?
**E.4 — Spot-check:** 3 случайных ACTIVE файла → ссылки ведут на существующие пути?
**E.5 — Root Cause Escalation** если найдены ошибки.

### Фаза F — Finalize

1. VERSION.md → обновить
2. SYSTEM_LOG → запись
3. PROGRESS → что сделано
4. CHANGELOG → какие файлы
5. Миграция_[date]/ → архив
6. Попросить пользователя сделать бэкап нового состояния

### Фаза G — Post-migration role-switch аудит

После финализации — независимый read-only аудит НЕ тем контекстом, что делал миграцию (Правило 16 п.4):
1. Роль: «профессиональный аудитор IT-систем» — в идеале отдельная сессия/экземпляр агента (свежий контекст), при невозможности — явное переключение роли.
2. Read-only по чистому состоянию (или клону): детерминированные проверки — `grep` старых паттернов = 0, `git branch -a` (ВСЕ ветки, не только HEAD — иначе пропустишь несмёрженные feature branches), сверка с checkpoint, spot-check ссылок.
3. Отчёт: AUDIT_REPORT в `System/working/` (вердикт 🟢/🟡/🔴 + список дефектов).
4. Дефекты найдены → Root Cause Escalation (Инвариант 7) → исправить → повторить аудит.

## ТИПЫ МИГРАЦИЙ

**Upgrade** (v[X] → v[Y]) — полный протокол.
**Repair** (v[X]-broken → v[X]-fixed) — фокус на повреждениях, только references.
**Restructure** (без смены версии) — полный протокол, VERSION patch.
