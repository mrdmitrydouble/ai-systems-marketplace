# AI Project Intelligence — Инструкция (v7)

## Что это

Это система управления проектами на базе AI (Claude от Anthropic). Она превращает Claude из одноразового чат-бота в **постоянного ассистента**, который:

- **Помнит** контекст между сессиями (кто, что, когда, зачем)
- **Трекает** обязательства (твои и чужие), напоминает о дедлайнах
- **Ведёт документацию** автоматически (прогресс, решения, наблюдения)
- **Учится** на ошибках и предлагает улучшения (с твоего согласия)
- **Масштабируется** — один мета-проект управляет множеством рабочих
- **Соединяется с другими узлами ризомы** через Git — улучшения автоматически циркулируют между партнёрами

## Откуда это пришло

Система создана в марте 2026 (проект «Создание ИИ систем», Дмитрий Рубин). На момент v6.3 — 40+ сессий, 9 проектов в эксплуатации, 12 итераций скилла. Теоретический фундамент:

| Фреймворк | Что даёт | Простыми словами |
|-----------|----------|------------------|
| **VSM** (Стаффорд Бир) | Жизнеспособность | Система как организм — выживает в меняющейся среде |
| **L1/L2** (Бейтсон) | Самообучение | L1 = делает задачи. L2 = извлекает уроки из опыта |
| **PID** (ТАУ) | Стабильность | Как термостат: уставка (CONSCIOUSNESS.md), обратная связь (аудит), система не дрейфует |
| **Холорархия** (Уилбер) | Вложенность | Каждый проект — и целое (сам по себе), и часть (мета-системы) |
| **Ризома** (Делёз) | Горизонтальные связи | Общий генетический код, но каждый проект мутирует по-своему |

Система распространяется через **споры** — скиллы, которые при запуске разворачивают полноценную систему. **Если ты получил эту спору от другого носителя ризомы — твой проект автоматически становится частью его экосистемы**, разделяя генетический код и обмениваясь уроками через Git.

## Что нового в v7 (главное обновление с v6.3)

v7 — итог трёх месяцев боевой эксплуатации на 9 проектах: конвергентные паттерны, которые независимо выработали несколько проектов, стали стандартом.

### 🔀 Параллельные сессии — штатный режим
Можно открывать несколько сессий по одному проекту одновременно. Страж параллельности (PreToolUse-хук) разводит записи в общую память по личным инбоксам сессий — затирание стало архитектурно невозможным. Слияние происходит автоматически при закрытии или старте следующей сессии. Тебе — ноль кнопок.

### 🔍 Независимый контур верификации (Правило 16)
Самопроверка ловит меньше половины собственных ошибок (боевая метрика: ~46%). Теперь стандарт: каждая цифра для внешних материалов — лично через первоисточник; регуляторные вопросы — адверсариальная проверка до фиксации; формулы — 3 сверки на сценариях; после крупных блоков — аудит свежим контекстом.

### ✍️ Документы без «ИИ-духа» (Правило 17)
Обязательная стадия «деИИзации» перед отправкой внешним адресатам: конкретный чек-лист маркеров ИИ-текста + жанр под адресата. Внешние читатели распознают и бракуют машинный стиль — это вопрос доверия.

### 📦 Git-дисциплина: коммит после логической единицы
«Сделано на диске» ≠ «сохранено». Боевой случай: готовый клиентский пакет 2+ недели висел незакоммиченным — одна команда очистки стёрла бы всё. Плюс git-workflow для team-проектов (Правило 19).

### 🩺 quick_health.sh и канон лимитов
Детерминированный health-скрипт при старте сессии: лимиты, целостность, git, здоровье самого harness. Все лимиты файлов — в одной канонической таблице (раньше один лимит жил в трёх версиях одновременно).

### 🧭 Вход после паузы
Самодиагностика трекеров после паузы >7 дней + «меню энергии» (10/30/120 минут) вместо вопросов «где ты был». Для scheduled-задач — heartbeat-принцип: нечего сказать — молчи.

### 📚 Новые шаблоны (опционально, по типу проекта)
KNOWN_ERRORS.md (реестр ошибок с разбором до системной причины; хронический паттерн → хук), FINDINGS_LOG + DECISIONS_LOG (аналитические проекты), PROJECT_PATHS.md (мета), модули правил: точность расчётов, двойной критик, engagement.

## Что нового в v6.3 (обновление с v6.2.2)

### 🛡️ Harness закрытия сессии — 4 слоя защиты

**Проблема:** Claude иногда объявляет «сессия завершена» БЕЗ фактического обновления памяти (PROGRESS/SYSTEM_LOG/HANDOFF). В реальной эксплуатации (проект ОКО) повторялось 7+ раз — каждый раз потеря контекста для следующей сессии.

**Решение** — детерминированная обвязка вокруг Claude:

1. **`/end` slash-команда** — позитивный путь: явный протокол закрытия с валидацией чеклиста
2. **Stop-hook** (`stop_guard.sh`) — ловит декларацию закрытия без `/end`, блокирует с инструкцией исправить
3. **SessionEnd-hook** (`session_end_snapshot.sh`) — при закрытии окна без прощания делает WIP-коммит + маркер `.session_incomplete.json`
4. **SessionStart-hook** (`session_start_recovery.sh`) — следующая сессия автоматически предлагает восстановить wrap-up

**Это работает в боях:** реальный кейс — две прерванные сессии за один день перехвачены без потери данных.

### 🧬 Ризоматический паспорт (полное соединение узлов)

Каждый проект получает уникальный паспорт в `VERSION.md`:
- `genesis_id` — UUID v4, твой узел (никогда не меняется)
- `parent_id` — UUID источника споры (или `null` если ты первичная ризома)
- `rhizome_name` — имя ризомы (одинаковое для всех узлов одной семьи)
- `origin_hash` — SHA-256 от CONSCIOUSNESS.md (видовой маркер)
- `genome_version` + `genome_hash` — версия генома и его хеш (мутирует при изменении [EVOLVABLE] зон)

**Зачем:** при встрече двух систем (даже через годы) ДНК-маркеры позволяют распознать «своих», обменяться уроками, восстановить родословную. На практике — Git push/pull между репозиториями участников ризомы.

### 🔍 Защита от галлюцинаций (Правило 1.8)

Агент явно различает «прочитано из файлов» (можно ссылаться) vs «внешний факт — нужна верификация», говорит «не знаю» вместо угадывания, привязывает утверждения к строкам файлов.

### 🔄 Обратная пропагация в мету (Правило 14)

Когда подопечный проект учит мету чему-то новому — мета явно проверяет «применимо ли это к мета самой?» ДО записи урока. Защита от паттерна «исправил подопечного, забыл про себя».

### 👥 Team sync (Правило 15)

В team-режиме при записи в чужой inbox или shared_board — немедленный `git commit + push`. Без этого «сообщение записано» ≠ «доставлено» (реальный случай: Юля 7+ дней молчала, потому что её агент не сделал push).

## Как установить (Claude Code + Git)

### Шаг 1 — Подготовь среду

```bash
# 1. Установи Claude Code (если ещё нет)
# https://claude.com/product/claude-code

# 2. Создай папку для МЕТА-проекта
# ВАЖНО: НЕ внутри Яндекс.Диска / iCloud / Dropbox — облачная синхронизация
# несовместима с агентом-писателем (проверено 70+ конфликтных копий в реале).
mkdir -p ~/Documents/Claude/Projects/[Название_твоего_проекта]
cd ~/Documents/Claude/Projects/[Название_твоего_проекта]

# 3. Запусти Claude Code
claude
```

### Шаг 2 — Установи скилл

В Claude Code:
- Settings → Skills → Add skill
- Укажи путь к распакованному `project-setup.zip`
- Перезапусти сессию (если потребуется)

### Шаг 3 — Запусти инициализацию

Напиши Claude:
> «Инициализируй мета-проект. Я получил спору от [имя_отправителя].»

Claude задаст 3-5 вопросов про твой проект, создаст структуру (`Project/` + `System/`), настроит память, поставит scheduled tasks (ежедневный + еженедельный аудит). Займёт ~30-45 минут.

### Шаг 4 — Соединись с ризомой отправителя

При вопросе «Это первичная ризома или ты получил спору?» — ответь **«Из споры»** и сообщи Claude:

- **parent_id** = `[GENESIS_ID отправителя]`
- **rhizome_name** = `[имя ризомы — наследуется]`
- **origin_hash** = `[hash отправителя — наследуется]`
- **genome_version** = `[версия на момент получения]`

(Эти данные приходят в файле `СПОРА_ОТ_*.md` вместе со скиллом.)

Твой `genesis_id` — новый UUID v4 (ты отдельный узел), но `rhizome_name` и `origin_hash` — общие с отправителем. Это и есть «принадлежность к одной ризоме».

### Шаг 5 — Установи Git-соединение с отправителем

```bash
# Создай приватный GitHub-репо для проекта
gh repo create [имя_репо] --private --source=. --push

# Пригласи отправителя collaborator-ом (он увидит твой узел)
gh api repos/[твой_username]/[имя_репо]/collaborators/[username_отправителя] -X PUT -f permission=push
```

После этого L2-уроки могут циркулировать между узлами через `git push/pull`. Отправитель тоже добавит тебя в свой репо при необходимости.

## Архитектура: твой узел в ризоме

```
                  ╔══════════════════════════╗
                  ║  Ризома "[имя ризомы]"   ║
                  ║  origin_hash: [...]      ║
                  ╚═══════════╤══════════════╝
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
   [Узел отправителя]   [Твой узел]           [Другие узлы]
   parent: null          parent: [отпр.]      parent: ...
        │                     │
   Подопечные: A, B, C   Подопечные: X, Y    ...

   Через Git push/pull: уроки и улучшения циркулируют между узлами
```

**Ты:**
- Самостоятельный узел (свой `genesis_id`, своя стратегия, свои подопечные)
- Часть общей ризомы (общий `rhizome_name`, `origin_hash`, генетический код)
- Можешь обмениваться уроками с отправителем через Git
- Можешь сам отправлять споры дальше — твои получатели будут «внуками» по родословной

## Team-режим (если работаешь с партнёрами на этом узле)

Если на твоём узле работают 2+ человек (например, ты + сотрудник):

1. **Один проект = один GitHub-репо** (приватный)
2. **Личные файлы памяти:** `PROGRESS_name.md`, `HANDOFF_name.md`, `SYSTEM_LOG_name.md`
3. **Notifications/:** `[имя]_inbox.md` для каждого + `shared_board.md` для общего
4. **При записи в чужой inbox** → немедленный `git commit + push` (Правило 15)
5. **При старте сессии** агент спрашивает: «Кто работает?»

Скилл сам всё это настроит, если на Шаге 1 онбординга ответишь «работаем вдвоём» или больше.

## Как пользоваться (ежедневно)

### Начало работы
Просто открой папку проекта в Claude Code и начни писать. Агент автоматически:
- Прочитает CLAUDE.md → STARTUP.md → ключевые файлы памяти
- `git pull` (синхронизация с другими узлами)
- Покажет приоритеты и просроченные обязательства
- Если прошлая сессия закрыта некорректно — предложит восстановить wrap-up

### Во время работы
- Скидывай файлы в `Project/Sources/` — агент проанализирует
- Говори что делать — агент классифицирует (S/M/L) и действует
- Обязательства фиксируются автоматически (🔴🟠🟡🔵)

### Завершение
Скажи **`/end`** или **«закончим»** — агент:
1. Обновит PROGRESS.md (что сделано)
2. Запишет наблюдения в SYSTEM_LOG.md
3. Создаст HANDOFF.md (контекст для следующей сессии)
4. Сделает `git commit + push`
5. Поставит флаг корректного закрытия

Если просто закроешь окно — SessionEnd-hook сам поймает и сделает WIP-коммит. Следующая сессия предложит восстановить.

### Какую модель выбрать
- **Базовая модель** — для рутины (80% задач)
- **Флагманская модель** — для сложного (стратегия, анализ нескольких файлов, ревью)

(Конкретные имена моделей не привязаны намертво — они обновляются каждые месяцы; выбирай актуальные «быструю» и «флагманскую» из доступных в твоей подписке.)

## Дорожная карта роста

### Этап 1 — Инициализация (ты здесь)
Мета-проект развёрнут. Система памяти работает. Соединён с ризомой отправителя.

### Этап 2 — Первые подопечные
Запускаешь 1-3 рабочих проекта (бизнес, продукт, отношения...). Мета-проект видит общую картину. Кросс-проектный аудит.

### Этап 3 — Экосистема
Scheduled tasks делают рутину автоматически. L2-уроки циркулируют между проектами. Система = единый организм.

### Этап 4 — Расширение ризомы
Соединение с узлами других носителей через Git. RAG для поиска по всем проектам. Возможно — собственные споры дальше своим партнёрам.

## FAQ для бизнес-партнёра

**Q: Я уже работаю с Claude / у меня есть свой Claude-проект. Что произойдёт?**
A: Скилл создаёт новый проект в новой папке. Существующие проекты не трогаются. Можно потом мигрировать их на эту систему — есть протокол `MIGRATION.md` в скилле.

**Q: Должен ли я обязательно соединяться с ризомой отправителя?**
A: Нет. Можешь развернуть как первичную ризому (`parent_id = null`). Но тогда теряется главное преимущество: автоматический обмен уроками через Git. Соединение — это всё что нужно для ризомы; никаких прав/контроля у отправителя над твоим узлом нет.

**Q: Что если у меня партнёры — нужно сразу настраивать team-режим?**
A: Можно начать solo и потом перевести в team (есть протокол `MIGRATION.md`). Если изначально знаешь — настрой сразу: на Шаге 1 онбординга агент спросит про команду.

**Q: Что если сессия прервалась?**
A: Harness v2 поймает: SessionEnd-хук сделает WIP-коммит + маркер. SessionStart следующей сессии предложит восстановить wrap-up из git log. Реально работает (проверено 30.04 на двух прерванных сессиях подряд).

**Q: Безопасно ли хранить чувствительные данные?**
A: Файлы хранятся локально на твоей машине. GitHub репо — приватный. Для особо чувствительного: `Project/Sources/private_*/` + правила приватности в RULES.md. `.gitignore` шаблон уже включает `*secret*`, `*.env`, `*.key`.

**Q: Что если я хочу изменить правила системы?**
A: Скажи агенту — он предложит изменение и подождёт твоего подтверждения. CONSCIOUSNESS.md (конституция системы) редактируешь только ты сам.

**Q: Можно ли использовать с другим AI (не Claude)?**
A: Ядро системы — markdown-файлы, переносимы. Но автозагрузка, scheduled tasks и harness специфичны для Claude Code. На другой LLM придётся переписать обвязку.

**Q: Сколько стоит?**
A: Подписка Anthropic Pro ($20/мес) — минимум. Для активной работы рекомендуется Max ($100/мес, включает Claude Code + большой контекст). Сама система бесплатна — open spore.
